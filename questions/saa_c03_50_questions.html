Design Secure Architectures

Identity, encryption, network protection, and the eternal exam favorites: roles over keys, deny over hope.

Q1SecureCross-account IAM access

An external audit firm (its own AWS account) needs read-only access to S3 buckets in your production account for a 6-week engagement. Security requires no long-term credentials to leave your account. What should you configure?

  1. Create IAM users for each auditor in your account and email them access keys.
  2. Create an IAM role in your account with an S3 read-only policy and a trust policy naming the auditor's account; auditors assume the role via STS.
  3. Make the buckets public for the engagement and share the URLs.
  4. Generate one shared access key pair and rotate it weekly during the audit.
Answer & reasoning
Correct: B

Cross-account access is the canonical use case for an IAM role assumed via STS. The trust policy limits who can assume it, the permissions policy limits what they can do, credentials are temporary (auto-expiring), and every AssumeRole call lands in CloudTrail. Delete the role when the audit ends.

A & D create long-lived keys — exactly what security prohibited, and keys can't be scoped by time. C is a data breach with extra steps.

Q2SecurePrivate S3 access — VPC endpoints

EC2 instances in private subnets must read/write S3. Compliance mandates that traffic never traverses the public internet and that the bucket only accepts requests originating from this VPC. Which design meets both requirements at no additional data-processing cost?

  1. Route S3 traffic through a NAT gateway in a public subnet.
  2. Create a Gateway VPC endpoint for S3, add it to the private route tables, and add a bucket policy condition on aws:sourceVpce.
  3. Create an Interface VPC endpoint for S3 in each AZ and update DNS.
  4. Attach an internet gateway and restrict the bucket by the instances' public IPs.
Answer & reasoning
Correct: B

Gateway endpoints for S3 are free, keep traffic on the AWS network, and pair with a bucket policy aws:sourceVpce / aws:sourceVpc condition to lock the bucket to your VPC — both requirements, zero data-processing charges.

A stays on AWS's network but hits the public S3 endpoint and bills NAT processing per GB. C works but charges hourly + per-GB — the question said no additional cost. D puts instances on the internet, violating requirement one.

Q3SecureS3 encryption choice — SSE-KMS

A healthcare app stores records in S3. Auditors require an audit trail of every use of the encryption key and the ability to control and rotate the key on the company's own schedule. Which encryption option satisfies this?

  1. SSE-S3 (Amazon-managed keys).
  2. SSE-C (customer-provided keys sent with each request).
  3. SSE-KMS with a customer managed KMS key.
  4. Client-side encryption with keys stored on the app servers.
Answer & reasoning
Correct: C

Only SSE-KMS with a customer managed key gives you both asks: every Encrypt/Decrypt call is logged to CloudTrail (the audit trail), and you own the key policy, rotation schedule, and lifecycle.

A has no key-level audit trail or rotation control. B means you transmit and manage raw keys yourself — no CloudTrail, high operational risk. D shifts all key management burden onto you with no managed auditing.

Q4SecureCredential storage & rotation

Lambda functions connect to an RDS PostgreSQL database. Security mandates the DB password be encrypted at rest, never in code, and rotated automatically every 30 days without redeploying functions. What is the best fit?

  1. Store the password in Systems Manager Parameter Store as a SecureString.
  2. Store the password in AWS Secrets Manager with a 30-day rotation schedule.
  3. Store the password in an encrypted environment variable per function.
  4. Store the password in an encrypted S3 object the functions download.
Answer & reasoning
Correct: B

Secrets Manager is purpose-built for this: KMS-encrypted storage plus native automatic rotation using a managed Lambda rotator for RDS — the 30-day rotation is a checkbox, and functions always fetch the current value at runtime.

A encrypts fine but has no built-in rotation — you'd build the rotator yourself. C requires a redeploy to rotate. D is DIY secret management with none of the rotation or fine-grained auditing.

Q5SecureSecurity groups vs network ACLs

Your team identifies a single malicious IP address hammering EC2 instances in one subnet. You must explicitly block that IP at the network layer while leaving all other traffic untouched. Where do you put the rule?

  1. Add a deny rule for the IP to the instances' security group.
  2. Add an inbound DENY rule for the IP to the subnet's network ACL, numbered before the allow rules.
  3. Create a Route 53 rule to drop queries from the IP.
  4. Enable VPC Flow Logs to reject the traffic.
Answer & reasoning
Correct: B

This is the classic stateful-vs-stateless split: security groups have no deny rules — they are allow-lists only. NACLs are stateless and support explicit DENY, evaluated in rule-number order, so a low-numbered deny for that IP blocks it subnet-wide.

A is impossible by design. C — Route 53 resolves DNS; it doesn't filter packets. D — Flow Logs record traffic; they never block anything.

Q6SecureCognito user pools vs identity pools

A mobile app needs users to sign up / sign in (including Google login) and then upload photos directly to an S3 bucket with temporary, scoped AWS credentials. Which combination delivers this with the least custom code?

  1. A Cognito user pool for authentication, plus a Cognito identity pool to exchange the token for temporary AWS credentials.
  2. A Cognito user pool alone — its tokens grant S3 access directly.
  3. Create an IAM user per app user and embed access keys in the app.
  4. A Cognito identity pool alone handling both sign-in and credentials.
Answer & reasoning
Correct: A

Two different jobs: the user pool is the directory — sign-up, sign-in, social federation — and issues JWTs. The identity pool exchanges those JWTs for temporary IAM credentials scoped by role, which is what S3 actually understands.

B — user-pool JWTs aren't AWS credentials; S3 can't consume them directly. C — an IAM user per end user is an anti-pattern and embedded keys leak. D — identity pools federate credentials but aren't a user directory with sign-up/sign-in UX.

Q7SecureAWS WAF placement

A web application behind an Application Load Balancer is being probed with SQL-injection payloads in query strings. You must block these requests before they reach the app, with minimal development effort. What do you deploy?

  1. AWS Shield Advanced on the ALB.
  2. Amazon GuardDuty with S3 protection enabled.
  3. An AWS WAF web ACL with the managed SQL-injection rule group, associated with the ALB.
  4. Stricter NACL rules on the web subnets.
Answer & reasoning
Correct: C

WAF inspects Layer-7 request content — exactly where SQLi lives — and the AWS-managed SQLi rule group means no rules to write. WAF attaches natively to ALB, CloudFront, and API Gateway.

A — Shield is DDoS protection (volumetric), not payload inspection. B — GuardDuty detects threats from logs; it blocks nothing. D — NACLs see IPs and ports, not HTTP query strings.

Q8SecureOrganizations SCPs

A company with 120 AWS accounts in AWS Organizations must guarantee that no one — including member-account root users — can disable CloudTrail. What is the enforceable control?

  1. Attach an identical IAM policy denying cloudtrail:StopLogging in every account.
  2. Attach a Service Control Policy denying cloudtrail:StopLogging and cloudtrail:DeleteTrail at the organization root/OU.
  3. Set IAM permission boundaries on all users and roles.
  4. Create an AWS Config rule that checks whether CloudTrail is enabled.
Answer & reasoning
Correct: B

SCPs are the only guardrail that constrains member-account root users. One deny policy at the root or OU level applies to all 120 accounts, is centrally managed, and cannot be removed from inside a member account.

A — root ignores IAM policies, and 120 copies drift. C — permission boundaries apply per-principal and never to root. D — Config is detective: it reports the trail was stopped; it doesn't prevent it.

Q9SecureEC2 credentials — instance profiles

An application on EC2 writes objects to S3 and items to DynamoDB. A security review flags hard-coded access keys in a config file on the instance. What is the recommended remediation?

  1. Move the keys into environment variables set at boot via user data.
  2. Encrypt the config file with KMS and decrypt it at startup.
  3. Attach an IAM role to the instance (instance profile) with least-privilege S3/DynamoDB permissions and delete the keys.
  4. Rotate the access keys every 24 hours with a cron job.
Answer & reasoning
Correct: C

Instance profiles deliver short-lived credentials through the instance metadata service; the SDKs pick them up automatically, they rotate themselves, and nothing sensitive sits on disk. This is the AWS-stated best practice for compute workloads.

A — user data is readable via the metadata/describe APIs; still static keys. B — an encrypted static key is still a static key with a new decryption problem. D — rapid rotation reduces exposure but keeps the fundamental anti-pattern alive.

Q10SecureAWS Config vs CloudTrail

Compliance needs to continuously record the configuration state of resources (e.g., "was this security group ever open to 0.0.0.0/0, and when?") and auto-flag violations against internal rules. Which service is the foundation?

  1. AWS CloudTrail with log file validation.
  2. AWS Config with configuration recorder and Config rules.
  3. AWS Trusted Advisor security checks.
  4. Amazon Inspector continuous scanning.
Answer & reasoning
Correct: B

Config is the "state and time machine": it snapshots each resource's configuration, keeps the change history, and evaluates Config rules (managed or custom) for compliance, with auto-remediation hooks.

A — CloudTrail answers who called which API, not what state a resource is in now or whether that state is compliant. C — a fixed best-practice checklist, not your custom rules or history. D — Inspector finds software vulnerabilities on workloads, not resource-configuration drift.

Q11SecureMacie vs GuardDuty vs Inspector

Legal suspects that S3 buckets across the account contain unprotected personally identifiable information (PII) — names, card numbers, credentials. You need a managed service to discover and classify this sensitive data at scale. Which one?

  1. Amazon GuardDuty
  2. Amazon Inspector
  3. Amazon Macie
  4. Amazon Detective
Answer & reasoning
Correct: C

Macie is the data-security service: it uses ML and pattern matching to find and classify sensitive data in S3 — PII, financial data, credentials — and reports where it lives and whether the bucket is exposed.

Memorize the trio: GuardDuty = threat detection from logs (accounts/network), Inspector = vulnerabilities in EC2/ECR/Lambda workloads, Macie = sensitive data in S3. Detective investigates findings after the fact.

Q12SecureEncrypting an existing EBS volume

An audit finds a production EC2 instance running on an unencrypted EBS volume. You must produce an encrypted volume containing the same data. What is the correct procedure?

  1. Open the volume settings and toggle encryption on.
  2. Snapshot the volume, copy the snapshot with encryption enabled, create a new volume from the encrypted copy, and swap it onto the instance.
  3. Enable "EBS encryption by default" in the region — existing volumes encrypt automatically.
  4. Attach the volume to KMS directly via a grant.
Answer & reasoning
Correct: B

Encryption cannot be switched on for an existing volume. The supported path is snapshot → copy the snapshot with encryption (choose your KMS key) → create the encrypted volume → detach/attach swap. Same data, now encrypted.

A — no such toggle exists. C — "encryption by default" affects only newly created volumes/snapshots. D — grants authorize key use; they don't retroactively encrypt storage.

Q13SecureSession Manager over bastions

Admins need shell access to EC2 instances in private subnets. Security wants zero inbound ports open, no SSH key management, and every session logged for audit. What is the best solution?

  1. A hardened bastion host in a public subnet with SSH key rotation.
  2. AWS Systems Manager Session Manager with the SSM agent and an instance role, logging sessions to CloudWatch/S3.
  3. A site-to-site VPN plus direct SSH to each instance.
  4. EC2 Instance Connect through an internet gateway.
Answer & reasoning
Correct: B

Session Manager checks every box: the agent makes outbound connections only (no inbound rules at all), access is governed by IAM instead of SSH keys, and full session transcripts stream to CloudWatch Logs or S3.

A — a bastion is an exposed host with open port 22 and keys to manage: exactly what was prohibited. C — still SSH keys and open instance ports inside the network. D — requires reachable instances and port 22; wrong for private subnets with zero inbound.

Q14SecureACM + CloudFront region rule

You are adding HTTPS with a custom domain (www.example.com) to a CloudFront distribution. Certificates must renew automatically at no cost. What do you do?

  1. Request a public ACM certificate in us-east-1 and attach it to the distribution.
  2. Request a public ACM certificate in the region closest to most users.
  3. Buy a certificate from a third-party CA and install it on the origin EC2 instances only.
  4. Use a self-signed certificate on the distribution.
Answer & reasoning
Correct: A

Two exam facts in one: ACM public certs are free and auto-renew, and for CloudFront the certificate must live in us-east-1 (N. Virginia) because CloudFront is a global service anchored there.

B — any other region's cert simply won't appear in the CloudFront dropdown. C — securing only the origin leaves the viewer connection without your domain's cert (and adds renewal toil). D — browsers reject self-signed certs; CloudFront won't serve your domain with one.

Q15SecureThird-party access — external ID

A SaaS monitoring vendor needs read access to CloudWatch in your account. AWS best practice for granting a third party cross-account access — while preventing the confused-deputy problem — is to:

  1. Create an IAM user for the vendor and share its access keys.
  2. Create a cross-account IAM role trusting the vendor's account, requiring a unique sts:ExternalId condition the vendor supplies on AssumeRole.
  3. Share your account's root credentials under NDA.
  4. Create a role trusting the vendor's account with no conditions — simpler is safer.
Answer & reasoning
Correct: B

The external ID is a shared secret in the trust policy's condition. Because the vendor must pass your unique ID when assuming the role, another customer of that vendor can't trick the vendor into using its access against your account — the confused-deputy defense.

A — long-lived keys held by an outsider. C — never leaves the building; root is unshareable, full stop. D — without the external ID condition, the role is exactly what the confused-deputy attack exploits.

Q16ResilientMulti-AZ web tier

A web app runs on one EC2 instance in one AZ behind nothing. After an AZ outage took it down, you must redesign for high availability with automatic recovery and no manual intervention. What is the minimal correct architecture?

  1. Resize to the largest instance type available.
  2. An Application Load Balancer targeting an Auto Scaling group with minimum 2 instances spread across at least two AZs.
  3. Two instances in the same AZ behind an ALB.
  4. Route 53 weighted records pointing at two standalone instances.
Answer & reasoning
Correct: B

HA on the exam = redundancy across AZs + health-checked routing + self-healing. The ALB stops sending to unhealthy targets; the ASG detects the failure and launches replacements — including into the surviving AZ.

A — vertical scaling keeps a single point of failure. C — the original outage was the AZ itself; same-AZ redundancy dies together. D — DNS TTL caching makes failover slow, and nothing replaces the dead instance.

Q17ResilientRDS Multi-AZ vs read replica

A single-AZ RDS MySQL database backs a payments app. The requirement: survive an AZ failure with automatic failover and no data loss, keeping the same connection endpoint. Read traffic is modest. What do you enable?

  1. Add a read replica in another AZ and promote it if the primary fails.
  2. Convert the instance to a Multi-AZ deployment.
  3. Take automated snapshots every 5 minutes.
  4. Add a read replica in another region.
Answer & reasoning
Correct: B

Multi-AZ replicates synchronously to a standby in another AZ and fails over automatically behind the same DNS endpoint — which is precisely "no data loss, no manual steps, same endpoint."

A & D — replicas are asynchronous (possible data loss) and promotion is a manual, endpoint-changing event. C — restoring a snapshot means minutes-to-hours of RTO and up to 5 minutes of lost writes.

Q18ResilientDecoupling with SQS

An e-commerce checkout writes orders synchronously to a processing tier. During flash sales, the processing tier is overwhelmed and orders are lost. You must ensure no order is ever dropped and the processing tier can work at its own pace. What is the fix?

  1. Vertically scale the processing tier's database.
  2. Publish orders to an SNS topic that invokes the processors directly.
  3. Place an SQS queue between the tiers; workers in an Auto Scaling group poll it, scaling on queue depth.
  4. Increase the web tier's retry count and timeout.
Answer & reasoning
Correct: C

A durable queue converts a synchronous dependency into an asynchronous buffer: bursts pile up safely in SQS (messages persist until deleted), and workers consume at sustainable speed — even auto-scaling on queue depth.

A — the bottleneck is coupling, not DB size; the next bigger spike loses orders again. B — SNS push delivery still hammers a struggling consumer in real time; no pull-paced buffer. D — retries amplify load during the spike, making the collapse worse.

Q19ResilientRoute 53 failover routing

A primary web stack runs in us-east-1. If it becomes unhealthy, users should automatically be served a static "status" page hosted on S3 + CloudFront. Which Route 53 configuration achieves this?

  1. Weighted routing: 90% primary, 10% S3 page.
  2. Latency-based routing between the two endpoints.
  3. Failover routing: primary record with a health check, secondary record pointing at the static site.
  4. Geolocation routing with a default record for the S3 page.
Answer & reasoning
Correct: C

Failover routing is the active-passive policy: Route 53 serves the primary while its health check passes and flips to the secondary when it fails. The others distribute traffic among healthy endpoints — a different job.

A sends 10% of users to the status page all the time. B optimizes latency, not availability, and would route to the "closer" endpoint constantly. D routes by user location — unrelated to failure state.

Q20ResilientAurora Global Database DR

A relational trading platform needs cross-region disaster recovery with RPO of about 1 second and RTO under a couple of minutes, while keeping the day-to-day database MySQL-compatible. Which option meets these targets?

  1. Nightly RDS snapshot copies to the DR region.
  2. An RDS MySQL cross-region read replica to promote during disaster.
  3. Aurora Global Database with a secondary region promoted on failover.
  4. DynamoDB global tables.
Answer & reasoning
Correct: C

Aurora Global Database replicates at the storage layer with typical lag under a second (RPO ≈ 1s) and promotes a secondary region in about a minute (RTO) — the only relational option here that hits both numbers.

A — RPO of up to 24 hours. B — binlog replication lags variably and promotion of a large MySQL replica can take many minutes. D — meets the numbers but isn't relational/MySQL-compatible, failing the stated constraint.

Q21ResilientS3 Cross-Region Replication

Regulation requires every object written to a compliance bucket in eu-west-1 to also exist, automatically, in a bucket in eu-central-1, including future writes, with different ownership possible per region. What do you configure?

  1. An S3 lifecycle rule transitioning objects to eu-central-1.
  2. S3 Cross-Region Replication with versioning enabled on both buckets.
  3. A nightly cron job running a sync command.
  4. S3 Transfer Acceleration on the source bucket.
Answer & reasoning
Correct: B

CRR is the managed, automatic answer: new objects replicate asynchronously to the destination region, ownership can be remapped, and the prerequisite to remember for the exam is versioning on both buckets.

A — lifecycle rules move objects between storage classes, never between regions. C — a nightly sync means up to 24h of exposure and a script you must babysit. D — acceleration speeds uploads to a bucket; it copies nothing anywhere.

Q22ResilientDR strategy selection

Management sets a disaster-recovery target of RTO in the low minutes for a business-critical app, but wants to avoid paying for a full-capacity duplicate of production. Which DR strategy fits?

  1. Backup and restore.
  2. Pilot light.
  3. Warm standby.
  4. Multi-site active-active.
Answer & reasoning
Correct: C

Warm standby keeps a scaled-down but fully functional copy running in the DR region; recovery is "scale it up + flip traffic," landing RTO in minutes at a fraction of full duplication cost.

A recovers in hours. B — pilot light keeps data replicated but core services off; provisioning them pushes RTO to tens of minutes. D meets the RTO but is exactly the full-capacity duplicate management refused to pay for.

Q23ResilientShared storage — EFS

A CMS runs on Linux EC2 instances in an Auto Scaling group across three AZs. All instances must read and write the same POSIX file system concurrently, and storage must grow automatically. Which service?

  1. A large gp3 EBS volume attached to all instances.
  2. Amazon EFS mounted on every instance via mount targets in each AZ.
  3. Instance store volumes synced with rsync.
  4. An S3 bucket mounted as the primary POSIX file system.
Answer & reasoning
Correct: B

EFS is the managed NFS answer: concurrent multi-instance, multi-AZ access, POSIX semantics, and elastic capacity with no provisioning.

A — EBS is a single-AZ block device; even Multi-Attach is same-AZ and needs a cluster-aware file system. C — instance store is ephemeral and syncing three AZs by hand is a consistency nightmare. D — S3 is object storage; mounts exist but it isn't a true POSIX file system for concurrent writes.

Q24ResilientKinesis vs SQS

A clickstream platform needs ingestion where multiple independent consumer apps read the same records, records can be replayed for 7 days, and per-user ordering is preserved. Which service fits?

  1. SQS standard queue.
  2. SQS FIFO queue.
  3. Kinesis Data Streams with user ID as partition key.
  4. SNS topic with email subscriptions.
Answer & reasoning
Correct: C

Three tells point at Kinesis Data Streams: multiple consumers reading the same data (a queue delivers each message once, to one consumer), replay (Kinesis retains data 24h–365 days; SQS deletes on consume), and ordering per key (partition key → shard order).

A — no ordering, no replay, competing consumers. B — ordered, but still consume-and-delete with no replay or fan-out reads. D — pub/sub push with no retention or replay.

Q25ResilientLambda failure handling

A Lambda function processes events asynchronously. Occasionally events fail after all automatic retries. The team must capture every failed event for later analysis and reprocessing. What do you configure?

  1. Increase the function timeout to 15 minutes.
  2. A dead-letter queue (SQS) or an on-failure destination for the function's async invocations.
  3. Write a try/catch that logs errors to CloudWatch.
  4. Set reserved concurrency to 1 so events process in order.
Answer & reasoning
Correct: B

For async invokes, Lambda retries twice; after that the event is discarded unless a DLQ or on-failure destination is set. Routing failures to SQS preserves the full event payload durably for inspection and replay.

A — timeouts don't rescue events that fail for other reasons, and exhausted retries still vanish. C — a log line isn't a re-drivable message; you lose the payload's durability guarantees. D — concurrency shaping doesn't capture anything.

Q26ResilientElastiCache engine choice

A gaming backend needs an in-memory store for sessions and a sorted-set leaderboard, with sub-millisecond reads, replication across AZs, and automatic failover if the primary node dies. Which configuration?

  1. ElastiCache for Memcached with multiple nodes.
  2. ElastiCache for Redis with cluster-mode replication group, Multi-AZ, and automatic failover.
  3. A single ElastiCache Redis node with daily backups.
  4. An RDS MySQL Multi-AZ instance with a sessions table.
Answer & reasoning
Correct: B

The feature list is a Redis fingerprint: sorted sets (leaderboards), replication + Multi-AZ + auto-failover, persistence options. Memcached is simpler by design and has none of these.

A — Memcached offers no replication, failover, or advanced data types; node loss = data loss. C — a single node is the opposite of failover; a backup restores minutes-old data. D — a relational DB delivers millisecond-plus latency and no in-memory data structures.

Q27ResilientStorage Gateway hybrid backup

An on-premises data center must back up file shares to AWS while apps keep using an NFS mount, with recently used files cached locally for low latency and everything durably stored as S3 objects. Which service?

  1. AWS DataSync running weekly transfer tasks.
  2. AWS Storage Gateway — S3 File Gateway.
  3. AWS Snowball Edge appliances rotated monthly.
  4. Direct S3 API integration in every application.
Answer & reasoning
Correct: B

S3 File Gateway is a hybrid bridge: apps see a standard NFS/SMB share, the gateway keeps a local cache of hot data, and files land in S3 as native objects — continuous, transparent, no app changes.

A — DataSync is a transfer engine for migrations/scheduled copies; it presents no ongoing file mount or cache. C — Snowball is offline bulk transport, not a continuous backup path. D — rewriting every app to speak S3 is maximal operational effort, the opposite of the ask.

Q28ResilientDynamoDB global tables

A global key-value session service must be active-active in three regions: users read and write locally with single-digit-millisecond latency, and data converges automatically across regions. Which datastore?

  1. Aurora Global Database.
  2. DynamoDB global tables.
  3. RDS MySQL with cross-region read replicas.
  4. ElastiCache Redis replicated by an application-level process.
Answer & reasoning
Correct: B

Global tables are multi-region, multi-writer replicas of a DynamoDB table with automatic, last-writer-wins convergence — the managed answer for active-active key-value at millisecond latency.

A & C — one writable primary region; other regions are read-only until failover, so "write locally everywhere" fails. D — hand-rolled cross-region cache replication is fragile and unmanaged; the exam always prefers the managed capability.

Design High-Performing Architectures

Cache close to users, pick storage by access pattern, and match the database engine to the query shape.

Q29PerformanceCloudFront + S3 global delivery

Users in Asia and South America report slow load times for images and video stored in an S3 bucket in us-east-1. You must cut global latency without duplicating the data or changing the application. What do you add?

  1. Enable S3 Transfer Acceleration on the bucket.
  2. Create a CloudFront distribution with the bucket as origin and serve content via CloudFront URLs.
  3. Replicate the bucket to every user region with CRR and route by geolocation.
  4. Move the application to larger EC2 instances.
Answer & reasoning
Correct: B

CloudFront caches at 400+ edge locations, so repeat requests never cross an ocean. One origin copy, no app rewrite, and lower S3 egress since edge hits don't touch the origin — every stated constraint satisfied.

A — Transfer Acceleration speeds uploads to S3, not downloads to viewers. C — works but duplicates storage per region and adds replication/routing complexity the question ruled out. D — the bottleneck is geography, not compute.

Q30PerformanceDynamoDB Accelerator (DAX)

A product-catalog DynamoDB table serves an extremely read-heavy, read-mostly workload. Latency must drop from single-digit milliseconds to microseconds with minimal application changes. What do you deploy?

  1. An ElastiCache Redis cluster with app-managed cache-aside logic.
  2. DynamoDB Accelerator (DAX) via the DAX SDK client.
  3. Provision much higher RCUs on the table.
  4. Convert the table to a global table.
Answer & reasoning
Correct: B

DAX is a write-through/read-through cache that is API-compatible with DynamoDB — swap the client, keep the code, get microsecond cache-hit reads.

A — Redis can hit microseconds too, but you must write and maintain cache-aside/invalidation logic — the "minimal changes" constraint kills it. C — more RCUs buys throughput, not lower latency; DynamoDB stays millisecond-class. D — global tables address geography, not per-request latency.

Q31PerformanceEBS volume type selection

A database on EC2 needs a 500 GB volume delivering a consistent 10,000 IOPS. Cost must be as low as possible while meeting the target. Which EBS volume type?

  1. gp2, sized to 500 GB.
  2. gp3 with 10,000 provisioned IOPS.
  3. io2 with 10,000 provisioned IOPS.
  4. st1 throughput-optimized HDD.
Answer & reasoning
Correct: B

gp3 decouples IOPS from size: baseline 3,000, provisionable to 16,000 — so a 500 GB gp3 volume can be dialed to exactly 10,000 IOPS at a fraction of io2's price. Cheapest option that meets spec wins.

A — gp2 ties IOPS to size (3 IOPS/GB): 500 GB ≈ 1,500 IOPS; you'd need ~3.3 TB just to buy performance. C — meets spec but costs more; reserve io2 for >16K IOPS or durability-critical needs. D — HDD tops out around 500 IOPS; wrong medium entirely.

Q32PerformancePlacement groups

A tightly coupled HPC simulation runs across 20 EC2 instances that exchange data constantly. You must minimize inter-node network latency and maximize throughput between them. Which placement strategy?

  1. A spread placement group.
  2. A partition placement group across 3 AZs.
  3. A cluster placement group in a single AZ with enhanced networking.
  4. Distribute the instances across three AZs for resilience.
Answer & reasoning
Correct: C

Cluster placement packs instances onto the same high-bisection network segment in one AZ — the lowest possible node-to-node latency and highest throughput, which is the entire game for tightly coupled HPC.

A — spread does the opposite: anti-affinity across hardware (max 7/AZ) for fault isolation. B — partition groups isolate replica sets for big-data systems, not latency. D — cross-AZ hops add latency; here performance explicitly outranks AZ resilience.

Q33PerformanceGlobal Accelerator vs CloudFront

A multiplayer game uses a custom UDP protocol to reach game servers behind Network Load Balancers in two regions. Players worldwide need lower latency, two static anycast IPs for allow-listing, and near-instant regional failover. Which service?

  1. Amazon CloudFront.
  2. AWS Global Accelerator in front of the NLBs.
  3. Route 53 latency-based routing.
  4. S3 Transfer Acceleration.
Answer & reasoning
Correct: B

Three tells: UDP (CloudFront is HTTP/S only), static anycast IPs (Global Accelerator's signature feature), and fast failover (endpoint health steering on the AWS backbone in seconds, no DNS caching in the way).

A — wrong protocol layer: CDN for HTTP content, not raw UDP flows. C — DNS answers get cached by resolvers, so failover is slow and IPs aren't static. D — S3 uploads only; unrelated.

Q34PerformanceAthena serverless queries

Analysts need to run occasional ad-hoc SQL over terabytes of CSV/JSON logs already sitting in S3. There is no budget for standing infrastructure; pay only when a query runs. Which approach?

  1. Load the data into a provisioned Redshift cluster.
  2. Import the files into RDS PostgreSQL nightly.
  3. Catalog the data with a Glue crawler and query in place with Amazon Athena.
  4. Spin up an EMR cluster with Hive for each analysis.
Answer & reasoning
Correct: C

Athena queries S3 in place — serverless, standard SQL, billed per TB scanned. A Glue crawler builds the table schema automatically. Zero infrastructure between queries is exactly the constraint.

A — a provisioned cluster bills 24/7 for occasional use. B — nightly ETL into an OLTP engine for TB-scale scans is slow, costly, and stale. D — EMR means cluster spin-up and management per session; "occasional ad-hoc" says serverless.

Q35PerformanceFSx family selection

A lift-and-shift brings Windows applications that require SMB file shares, NTFS ACLs, and Active Directory integration. Which AWS storage service do they mount?

  1. Amazon EFS.
  2. Amazon FSx for Windows File Server.
  3. Amazon FSx for Lustre.
  4. S3 with File Gateway.
Answer & reasoning
Correct: B

The keyword triple SMB + NTFS + AD maps one-to-one onto FSx for Windows File Server — a fully managed native Windows file system.

A — EFS is NFS for Linux; no SMB, no NTFS ACLs. C — Lustre is the HPC/ML scratch file system; hearing "machine learning / hundreds of GB/s" is its cue, not Windows shares. D — File Gateway serves S3-backed shares for hybrid caching, not a native NTFS/AD experience.

Q36PerformanceRDS Proxy connection pooling

Hundreds of concurrent Lambda functions open connections to RDS PostgreSQL, and the database keeps hitting too-many-connections errors during bursts. Writes and reads both occur. What is the managed fix?

  1. Move to a much larger DB instance class.
  2. Add read replicas and point Lambdas at them.
  3. Put Amazon RDS Proxy between Lambda and the database.
  4. Cache all queries in ElastiCache.
Answer & reasoning
Correct: C

The failure mode is connection churn, not query load. RDS Proxy pools and multiplexes thousands of ephemeral Lambda connections onto a small warm set of DB connections, absorbing bursts and even speeding failover.

A — buys a higher connection ceiling until the next burst; costs scale badly. B — replicas are read-only; the errors include writers. D — caching helps reads but doesn't stop hundreds of functions from opening connections.

Q37PerformanceS3 Transfer Acceleration

Field teams on several continents upload multi-gigabyte video files to one central S3 bucket in us-east-1. Uploads over the public internet are slow and flaky. What speeds them up with minimal change?

  1. Enable S3 Transfer Acceleration and use multipart upload against the accelerated endpoint.
  2. Put CloudFront in front of the bucket.
  3. Ship Snowball devices from each site weekly.
  4. Create one bucket per continent and merge nightly.
Answer & reasoning
Correct: A

Transfer Acceleration routes uploads into the nearest CloudFront edge and carries them to the bucket on the AWS backbone, skipping the messy public-internet middle mile; multipart upload adds parallelism and resumability for large files. One endpoint change.

B — CloudFront is a download-side CDN; TA is the upload-optimized product built on the same edges. C — days of latency for a continuous workflow. D — multiplies buckets and adds a merge pipeline: complexity, not speed.

Q38PerformanceRedshift for OLAP

BI teams run complex aggregations joining billions of rows across sales, inventory, and customer tables — queries that take hours on the RDS MySQL production replica. Which datastore is designed for this workload?

  1. A larger RDS MySQL instance.
  2. Amazon Redshift (columnar data warehouse).
  3. DynamoDB with secondary indexes.
  4. Amazon Neptune.
Answer & reasoning
Correct: B

Scanning and aggregating billions of rows is OLAP: Redshift's columnar storage, compression, and massively parallel processing read only the needed columns across many nodes — the architecture built for exactly these joins.

A — row-based OLTP engines read whole rows; bigger hardware shifts, not solves, the mismatch. C — DynamoDB has no joins or ad-hoc aggregation. D — Neptune is a graph database for relationship traversal, not tabular analytics.

Q39PerformanceSNS → SQS fan-out

When an order is placed, three independent systems — invoicing, shipping, analytics — must each receive and process the event at their own pace, with failures in one never affecting the others. What is the pattern?

  1. One SQS queue with all three systems polling it.
  2. An SNS topic with three SQS queue subscriptions — one per system.
  3. The order service calls each system's API in sequence.
  4. A Kinesis stream with a single shared consumer application.
Answer & reasoning
Correct: B

Fan-out: publish once to SNS; each subscribed SQS queue gets its own durable copy, so every system consumes independently, buffered against its own outages, with per-queue DLQs and filtering available.

A — one queue means competing consumers: each message is processed by exactly one system, not all three. C — synchronous chaining couples all three; one failure blocks the order path. D — workable but a single shared consumer app recouples the teams; the classic decoupled answer is SNS + SQS.

Q40PerformanceAuto Scaling policy mix

A trading dashboard's traffic spikes sharply every weekday at 9:00 AM when markets open, then varies unpredictably through the day. Reactive scaling alone arrives too late for the opening bell. What policy combination fits?

  1. Target tracking on average CPU only.
  2. A scheduled scaling action pre-warming capacity before 9:00 AM, plus target tracking for the rest of the day.
  3. Manual scaling by the on-call engineer each morning.
  4. A step scaling policy with tighter CPU thresholds.
Answer & reasoning
Correct: B

Known, time-based demand → scheduled scaling (capacity is already warm at 8:45); unknown intra-day demand → target tracking. The exam loves this combination because each policy covers the other's blind spot.

A & D — any reactive policy needs the spike to happen first, then minutes of launch time: too late for a cliff-edge open. C — human-scheduled scaling is what scheduled actions automate away.

Design Cost-Optimized Architectures

Right storage class, right purchase model, and never pay a NAT gateway to reach S3.

Q41CostS3 lifecycle design

Application logs land in S3. Access pattern is known: heavy analysis for 30 days, rare lookups until day 90, then retained 7 years purely for audit (retrieval within 12 hours is acceptable). Design the lowest-cost storage plan.

  1. Keep everything in S3 Standard for 7 years for simplicity.
  2. Lifecycle rule: Standard → Standard-IA at 30 days → Glacier Deep Archive at 90 days → expire at 7 years.
  3. Use S3 Intelligent-Tiering for the entire lifespan.
  4. Lifecycle rule moving objects to Glacier Deep Archive on day 1.
Answer & reasoning
Correct: B

The pattern is known and time-based, so a deterministic lifecycle wins: IA cuts storage cost ~45% once reads become rare, and Deep Archive (cheapest class; standard retrieval ~12h — within the stated tolerance) carries the 7-year audit tail. Expiration ends the spend on schedule.

A — pays hot-storage prices for 7 years of cold data. C — Intelligent-Tiering earns its monitoring fee when patterns are unknown; here you'd pay it to discover what you already told it. D — 30 days of heavy analysis against a 12-hour-retrieval archive class breaks the workload (and retrieval fees explode).

Q42CostSpot for interruptible work

A nightly video-transcoding pipeline is stateless, checkpointed, and fully fault-tolerant — any interrupted job simply re-queues. Runs take 2–6 hours with no deadline pressure. Which purchase option minimizes compute cost?

  1. On-Demand instances.
  2. 3-year Standard Reserved Instances.
  3. Spot Instances (diversified across instance types/pools).
  4. Dedicated Hosts.
Answer & reasoning
Correct: C

The scenario recites the Spot checklist — stateless, checkpointed, interruption-tolerant, flexible timing — and Spot pays for it with up to ~90% off On-Demand. Diversifying pools softens interruption risk further.

A — full price for a workload that doesn't need the guarantee. B — RIs suit steady 24/7 baselines, not a nightly batch window. D — the most expensive option, for licensing/compliance needs that aren't present.

Q43CostIntelligent-Tiering

A media archive holds millions of objects whose access pattern is completely unpredictable — some files go viral years later, others are never touched. Finance wants automatic savings with no retrieval fees and no performance surprises. Which storage class?

  1. S3 Standard-IA for everything.
  2. S3 Intelligent-Tiering.
  3. S3 Glacier Flexible Retrieval.
  4. Manually crafted lifecycle rules per folder.
Answer & reasoning
Correct: B

Intelligent-Tiering is built for unknown patterns: it moves each object between access tiers based on observed use, charges a small monitoring fee, and — the clincher — has no retrieval fees and no minimum-duration penalty on the default tiers, so a viral spike costs nothing extra.

A — IA charges per-GB retrieval and a 30-day minimum: unpredictable access makes those fees unpredictable too. C — minutes-to-hours retrieval breaks "no performance surprises." D — hand-written rules require the very knowledge the scenario says you lack.

Q44CostKill the NAT data charge

Cost Explorer shows large NAT gateway data-processing charges. Investigation reveals private-subnet EC2 instances pushing terabytes to S3 in the same region through the NAT gateway. What change eliminates most of this cost?

  1. Replace the NAT gateway with a larger one for volume pricing.
  2. Add a Gateway VPC endpoint for S3 to the private subnets' route tables.
  3. Add a second NAT gateway to split the load.
  4. Create an Interface VPC endpoint for S3.
Answer & reasoning
Correct: B

The gateway endpoint carries S3-bound traffic directly over the AWS network with no hourly charge and no per-GB charge — the terabytes stop touching the NAT entirely, so the data-processing line item collapses.

A & C — NAT pricing has no volume discount and two NATs double the hourly cost; per-GB charges continue either way. D — an interface endpoint works but bills hourly + per-GB; for pure S3 cost elimination, gateway wins.

Q45CostSavings Plans selection

A company has a steady compute baseline spread across EC2 instances of changing families in three regions, Fargate tasks, and Lambda. They'll commit spend for 3 years but need freedom to keep re-architecting. Which commitment fits?

  1. Standard Reserved Instances for the current instance types.
  2. EC2 Instance Savings Plans in each region.
  3. Compute Savings Plans.
  4. Spot Instances for the entire baseline.
Answer & reasoning
Correct: C

Compute Savings Plans commit to $/hour of compute and automatically apply across any EC2 family, size, region, OS — plus Fargate and Lambda. Maximum flexibility for a shape-shifting architecture, still deep discounts.

A — Standard RIs lock family/region; every re-architecture strands the commitment. B — EC2 Instance SPs are cheaper per-hour but locked to a family+region and cover EC2 only — no Fargate/Lambda. D — Spot is interruptible; a steady always-on baseline can't live on it.

Q46CostServerless for idle-heavy APIs

An internal API receives a few thousand requests per day at unpredictable times, each finishing in under a second. It currently runs on two always-on EC2 instances that sit ~99% idle. What is the most cost-effective re-platform?

  1. Downsize to one smaller EC2 instance.
  2. API Gateway + Lambda, paying per request.
  3. An ECS cluster on EC2 with two services.
  4. Elastic Beanstalk with a single always-on environment.
Answer & reasoning
Correct: B

Sparse, short, unpredictable invocations are the serverless sweet spot: with Lambda + API Gateway you pay per request and per ms of execution — idle time costs zero. A few thousand sub-second calls a day lands near the free tier.

A — smaller idle is still 24/7 billing, plus a new single point of failure. C & D — both keep always-on instances underneath; the 99%-idle problem survives the re-platforming.

Q47CostEgress cost via CloudFront

A media site serves images directly from S3 to global users. The bill shows heavy S3 data-transfer-out (egress) and request charges. Without moving the data, how do you cut delivery cost and improve speed?

  1. Serve through a CloudFront distribution with the bucket as origin.
  2. Compress images and keep serving directly from S3.
  3. Move the bucket to the cheapest region.
  4. Switch the objects to Standard-IA.
Answer & reasoning
Correct: A

Two economics work in CloudFront's favor: S3 → CloudFront origin transfer is free, and cached edge hits never generate S3 requests at all — you pay CloudFront's (generally cheaper, cache-discounted) egress instead of raw S3 egress. Users get edge latency as a bonus.

B — helps a little, but every byte still exits S3 at S3 rates. C — regional price deltas are marginal and now everyone is far from the data. D — IA cuts storage cost but adds per-GB retrieval fees on a frequently read workload: the bill goes up.

Q48CostCost visibility & attribution

Finance must attribute AWS spend to departments and projects across a multi-account organization and let team leads slice it themselves. What is the foundational setup?

  1. Consolidated billing via Organizations, enforce resource tagging, activate cost allocation tags, analyze in Cost Explorer / CUR.
  2. Export CloudWatch metrics to a spreadsheet monthly.
  3. Create one IAM group per department.
  4. Ask each team to screenshot its bill.
Answer & reasoning
Correct: A

The chain is: Organizations rolls all accounts into one bill → tags (e.g., Department, Project) label resources → activating cost allocation tags makes those tags billing dimensions → Cost Explorer / Cost & Usage Report group and filter by them. Add AWS Budgets for proactive alerts.

B — CloudWatch measures performance, not dollars. C — IAM organizes permissions; it is invisible to billing. D — not a system.

Q49CostAurora Serverless for spiky dev DBs

A development team's relational database is used a few unpredictable hours per day and sits idle otherwise — but must be instantly available whenever someone connects, with no capacity management. What minimizes cost?

  1. A provisioned RDS instance stopped and started by cron scripts.
  2. Aurora Serverless, scaling with demand and down to minimal/zero capacity when idle.
  3. A provisioned RDS instance sized for peak, running 24/7.
  4. DynamoDB on-demand mode.
Answer & reasoning
Correct: B

Aurora Serverless bills by capacity units actually consumed, scaling up on connection demand and down toward zero when idle — intermittent + unpredictable + hands-off is its exact design brief.

A — cron windows and "unpredictable hours" contradict each other; RDS stop also auto-restarts after 7 days, and cold starts take minutes. C — pays peak price for ~90% idle. D — the team needs a relational database; on-demand pricing doesn't change the engine mismatch.

Q50CostGlacier retrieval tiers

Compliance archives live in S3 Glacier Flexible Retrieval. Auditors occasionally request batches of documents and accept delivery any time within 12 hours. Which retrieval tier keeps these requests cheapest?

  1. Expedited retrieval (1–5 minutes).
  2. Standard retrieval (3–5 hours).
  3. Bulk retrieval (5–12 hours).
  4. Move the archive to S3 Standard so retrieval is instant.
Answer & reasoning
Correct: C

Retrieval tiers trade speed for price; Bulk is the cheapest (large restores at the lowest per-GB rate) and its 5–12-hour window sits exactly inside the auditors' 12-hour tolerance. Pay for the speed you need — never more.

A — minutes-fast at the highest price for a deadline measured in hours. B — meets the deadline but costs more than Bulk for no required benefit. D — hot storage 24/7/365 to serve occasional audits inverts the entire cost model.
